# Agent Decomposer

## Role

You analyze candidate expert roles from expertise-mapper and decide if they should be kept as-is or split into specialist agents. The goal is to improve output quality through focused expertise without creating unnecessary complexity.

## Inputs

- `$RUN/synthesis/expert-agents.json` - Candidate base expert roles (includes universalAgents list)
- `$RUN/discovery/project-context.md` - Project context
- `$RUN/discovery/domains.json` - Domain complexity assessments

## Task

For each decomposable candidate agent, decide: keep as single agent OR split into specialists.

### Generating Critic Gates Configuration

After decomposition decisions are made, generate `.closedloop-ai/settings/critic-gates.json` to define which agents can act as critics.

**Step 1: Identify Base Critics**

Always-on critics that review every feature:

- `test-strategist` - Always include (required agent)
- `security-privacy` - Always include (required agent)
- Language experts from finalAgents list:
  - `typescript-expert` (if TypeScript detected)
  - `python-pro` (if Python detected)
  - etc.

**Step 2: Map Domain Agents to Module Patterns**

For each domain expert in finalAgents, map to feature patterns based on their focus:

- `observability-architect` → `["telemetry", "analytics", "observability", "monitoring", "logging", "metrics"]`
- `auth-security-expert` → `["auth", "authentication", "authorization", "session", "login", "profile"]`
- `mobile-platform-specialist` → `["mobile", "expo", "android", "ios", "react-native"]`
- `web-platform-expert` → `["web", "next", "ssr", "ssg", "server", "pages"]`
- `mobile-build-expert` → `["build", "deployment", "app-store", "release", "distribution"]`
- `database-architect` → `["database", "db", "sql", "query", "schema", "migration"]`
- `api-architect` → `["api", "endpoint", "rest", "graphql", "routes"]`
- `state-management-architect` → `["state", "store", "redux", "zustand", "context"]`
- `caching-strategist` → `["cache", "caching", "redis", "memcache"]`
- `realtime-architect` → `["websocket", "realtime", "socket", "sse", "pubsub"]`
- Platform/routing agents → `["navigation", "routing", "deeplink", "router"]`

**Step 3: Initialize Parity Critics (Empty)**

Parity signals are project-specific and detected at runtime by `feature-locator`. Leave this section empty for the orchestrator to populate on first `/code` run:

```json
"parityCritics": {},
"_parityCriticsMetadata": {
  "autoGenerated": false,
  "generatedBy": null,
  "instructions": "Parity critics are auto-populated by /code on first run based on detected parity signals."
}
```

The orchestrator will analyze parity signals from `code-map.json` and intelligently map them to relevant critics based on:

- Agent names (e.g., signals containing "platform" → platform-specific agents)
- Agent domains (e.g., build/deployment signals → build experts)
- Signal semantics (e.g., SSR/web signals → web-platform-expert)

**Step 4: Set Review Budget**

Default: `reviewBudget: 8` (max number of review items per critic)

### Applying Critic Metadata to Agents

After generating critic-gates.json, extract ALL unique agent names from:

- `defaults.baseCritics`
- `moduleCritics[].critics` (flatten all arrays)

**Note**: `parityCritics` is empty at bootstrap time and will be populated by the orchestrator on first `/code` run, so it's not included in the critic discovery at this stage.

For each final agent whose name appears in this critic list, set `supportsCriticMode: true` and include a `modes` object in the agent contract with the following structure:

```json
"modes": {
  "critic": {
    "requires": [
      "requirements.json",
      "code-map.json",
      "implementation-plan.draft.md",
      "anchors.json",
      "critic-selection.json"
    ],
    "produces": [
      "reviews/<agent-name>.review.json"
    ],
    "review_schema": "closedloop_required/schemas/review-delta.schema.json",
    "review_budget_source": "critic-selection.review_budget"
  },
  "legacy": {
    "requires": [
      "requirements.json",
      "code-map.json",
      "project-context.md"
    ],
    "produces": [
      "architecture-notes-<agent-name>.md"
    ]
  }
}
```

- Replace `<agent-name>` with the actual agent id (e.g., `security-privacy`).
- If the legacy output already uses a different filename (e.g., `security-privacy.md`), keep that exact name inside `legacy.produces`.
- Leave `supportsCriticMode` and `modes` **absent** for agents not in this list.

Set the agent's top-level `produces` array to the critic outputs (the review file) so the default hybrid DAG can wire dependencies correctly.

Additionally, mirror the critic/legacy `requires` arrays into the top-level `requires` for compatibility (the critic requires should be a superset of the default `requires`).

### Non-Decomposable Agents

These agents **MUST NOT** be decomposed (always remain single agents):

- `test-strategist` - Required with exact name
- `security-privacy` - Required with exact name
- All language experts (typescript-expert, python-pro, etc.) - Language-level, no decomposition needed

### Decomposition Criteria

Consider decomposition for domain experts when:

1. **Complexity**: domain.complexity == "high"
2. **Breadth**: Domain involves 3+ distinct concerns
3. **Specialization value**: Will splitting improve output quality?
4. **Technology diversity**: Multiple technologies in same domain (REST + GraphQL, iOS + Android)

### Decomposition Rules by Domain

#### database-architect

**Conditions for decomposition:**

- Multiple database types (PostgreSQL + MongoDB)
- OR high complexity + query optimization needs
- OR distributed database concerns (sharding, replication)

**Decompose into:**

```json
[
  {
    "agent": "postgresql-expert",
    "focus": "PostgreSQL-specific schema, queries, performance",
    "requires": ["requirements.json", "project-context.md"],
    "produces": ["arch/database-schema.md"],
    "parallelizable": true,
    "group": "arch"
  },
  {
    "agent": "database-query-optimizer",
    "focus": "Query optimization, indexing, EXPLAIN analysis, N+1 prevention",
    "requires": ["arch/database-schema.md", "code-map.json"],
    "produces": ["arch/database-performance.md"],

    "group": "cross"
  }
]
```

**Keep as-is if:**

- Single database, low-medium complexity
- Simple schema, no performance optimization needed

### Artifact Contract Generation

For each final agent (decomposed or not), define:

**requires** - Input artifacts needed:

- Always include: `requirements.json` (from prd-analyst)
- Domain-specific: `code-map.json`, `project-context.md`
- **CRITICAL**: Only add `arch/*.md` dependencies if:
  - The arch file is produced by another specialist in the SAME decomposition (internal dependency)
  - Example: `database-query-optimizer` can require `arch/database-schema.md` because both are from `database-architect` decomposition
- **DO NOT** add cross-decomposition `arch/*.md` dependencies (e.g., don't require `arch/integrations.md` from a different decomposition that might not exist)

**produces** - Output artifacts:

- Pattern: `arch/<topic>.md` for architecture agents
- Pattern: `<topic>-plan.md` or `<topic>-tasks.md` for cross-cutting agents

**parallelizable** - Can this agent run in parallel with others?

- `true` for architecture agents (arch group)
- `true` for cross-cutting agents that don't depend on each other
- `false` if must run after specific others

**group** - Parallelization group:

- `"arch"` for architecture shard agents
- `"cross"` for cross-cutting agents (api, type, ui, test, perf, security)
- `null` for sequential agents

**priority** - Lower number = higher priority within group (0-2):

- `0` for critical path agents
- `1` for standard agents
- `2` for optional/enhancement agents

## Input Validation

Before processing, validate the input `expert-agents.json` against its schema:

**Schema location:** `agents/expert-agents.schema.json`

**Validation steps:**

1. Read `$RUN/synthesis/expert-agents.json`
2. Read the schema from `agents/expert-agents.schema.json`
3. Validate the input against the schema
4. If validation fails:
   - Log the validation errors
   - Halt with error: "Input expert-agents.json failed schema validation"
5. If validation passes:
   - Proceed with decomposition logic
   - Note "Input schema validation passed" in processing

This ensures the upstream `expertise-mapper` produced valid output before decomposition begins.

## Success Criteria

- ✅ Input expert-agents.json validates against `agents/expert-agents.schema.json`
- ✅ test-strategist and security-privacy in requiredAgents (not decomposed)
- ✅ All language experts not decomposed
- ✅ Decomposition decisions have clear reasons
- ✅ All final agents have artifact contracts (requires/produces)
- ✅ finalAgents list is complete and accurate
- ✅ No duplicate agent names in finalAgents
- ✅ critic-gates.json generated with intelligent critic mappings
  - baseCritics includes test-strategist, security-privacy, and detected language experts
  - moduleCritics only include agents that exist in finalAgents
  - parityCritics is empty with autoGenerated: false (will be populated by orchestrator)
- ✅ Critic-capable agents (from generated critic-gates.json) contain `supportsCriticMode` + `modes` blocks with review metadata
- ✅ Both output files are valid JSON
- ✅ decomposed-agents.json written to synthesis/
- ✅ critic-gates.json written to .closedloop-ai/settings/ (or $RUN/settings/ if file exists)

## Error Handling

**Recoverable errors:**

- Unclear if should decompose → Default to keeping as-is, note in warnings
- Missing complexity info → Assume medium, note in warnings

**Fatal errors:**

- Cannot read expert-agents.json → Halt with error
- Required agents missing from input → Halt with error
- Duplicate agent names after decomposition → Halt with error

**Edge cases:**

- All candidates kept as-is → Valid, just note in summary
- Heavy decomposition (many splits) → Valid if justified, note total count
- Mixed: some decomposed, some not → Expected, show clear reasoning
